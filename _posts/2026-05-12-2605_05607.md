# MoE 训练被卡通信了？DySHARP 用交换机内计算直接干掉 50% 冗余流量

> 原文：[Accelerating MoE with Dynamic In-Switch Computing on Multi-GPUs](https://arxiv.org/abs/2605.05607)（ISCA 2026）

---

## 1. 前言吐槽

你有没有想过，训练 DeepSeek-V3 这种 MoE 模型时，GPU 的通信线路上到底跑了多少无用数据？

结果有点细思极恐——**近 50% 的通信流量是纯冗余的**，白白占着带宽，什么有价值的事也没干。

更离谱的是，NVIDIA 其实在 Hopper 时代就搞了一个叫 NVLink SHARP（NVLS）的东西，本来可以在交换机里做计算来消掉这些冗余，但结果因为 MoE 的通信模式是"动态的"，而 NVLS 只支持"静态的"，所以直接 GG。

这篇 ISCA 2026 的工作（DySHARP）就是来填这个坑的。今天想和大家聊聊这篇工作到底解决了什么真问题，以及它的两个核心设计有多精妙。

---

## 2. MoE 的通信瓶颈：到底卡在哪

先交代下背景。MoE（Mixture-of-Experts）的精髓是稀疏激活：每个 token 只走 top-k 个 expert，计算量大幅减少。但在多 GPU 训练时，expert 分散在不同 GPU 上（expert parallelism，EP），每个 token 就要经过两步 inter-GPU 通信：

- **Dispatch**：把 token 发送到对应 expert 所在的 GPU
- **Combine**：把各 expert 的输出汇总回 token 所在的 GPU

如下图，MoE EP 的整体执行流程，以及 Dispatch/Combine 引入的冗余：

![MoE EP 执行流程与通信冗余](assets/fig1_opportunity.png)

图中下半部分说的很清楚：

- **Dispatch 冗余**：Token B 需要发给 GPU2 和 GPU3，GPU0 就要把同一份数据传两次，走两条链路
- **Combine 冗余**：GPU2 和 GPU3 各自算出的中间结果，要分别传回 GPU0，这些数据本来可以在交换机里直接聚合

在 DeepSeek-V3 的实际测试里，**通信耗时占 MoE layer 执行时间的 70.4%**。而且随着 Blackwell、Rubin 算力越来越强，GPU 算得越来越快，通信瓶颈反而越来越突出。

## 4. NVLS 为什么搞不定

NVIDIA 其实早就想到了这个思路——NVLink SHARP（NVLS）就支持两种 in-switch computing：

- **in-switch multicast**：GPU 只传一次，交换机复制分发给多个目标，消除 Dispatch 冗余
- **in-switch reduction**：多个 GPU 的中间结果在交换机内聚合，只传最终结果回来，消除 Combine 冗余

但 NVLS 的设计只能处理**静态**通信模式，有两个硬约束：

- **Fixed target sets**：所有 token 都和同一组固定的 GPU 通信
- **Symmetric addressing**：token 在每个 GPU 上的内存偏移量完全一致

MoE 的 Dispatch/Combine 偏偏是**动态**的——Token A 可能发往 GPU{0,1,2}，Token B 发往 GPU{2,3}，目的地各不同；而且每个 GPU 上的内存 layout 是本地管理的，offset 也不一致。

如下图，静态集合通信 vs MoE 动态通信的模式对比：

![静态通信 vs 动态通信模式对比](assets/fig3_multimem.png)

那如果强行把 MoE 通信转成 AllGather/Reduce-Scatter 来套 NVLS 呢？得给所有 GPU 都发，包括根本用不上这个 token 的 GPU，引入大量无用流量。实测在 DeepSeek-V3 + GH200 NVL32 上，这个 workaround 引入了 **340% 的无用流量**，把 in-switch computing 省下来的那点好处完全冲掉了。

## 5. DySHARP 的核心思路：两个技术，缺一不可

这篇工作提出了 DySHARP，用两个技术配合解决问题：

**① Dynamic Multimem Addressing**：让交换机支持动态通信，消掉冗余流量
**② Token-Centric Kernel Fusion**：让 Dispatch 和 Combine 并发执行，把流量削减真正转化成速度

为什么两个缺一不可？先看下图（非常关键，建议认真看）：

![DySHARP 两技术组合的必要性分析](assets/fig4_motivation.png)

图中从上到下是 6 种配置的时间线：

- **(a) DeepEP（无 overlap）**：Dispatch 跑完再跑 GEMM，再跑 Combine，通信是瓶颈
- **(b) COMET（有 overlap）**：Dispatch-GEMM-1 和 GEMM-2-Combine 各自 overlap，但通信量没减
- **(c) DySHARP-Basic**：加了 dynamic multimem addressing，GPU→Switch 流量减了，但 Switch→GPU 方向没减，不对称，加速上不去
- **(d) DySHARP-COMET**：基础 overlap + in-switch computing，依然受不对称拖累
- **(e) Kernel Fusion Only**：只做 Dispatch/Combine 并发，没有流量削减，相比 COMET 没有提升
- **(f) DySHARP 完整方案**：两者结合，Dispatch（GPU→Switch 主导）和 Combine（Switch→GPU 主导）并发执行，互补的非对称流量 merge，带宽利用率大幅提升 ✅

**核心洞察**：in-switch computing 减的是"单向"流量，天生不对称；kernel fusion 让两个方向并发，把互补的非对称流量叠加，才能真正转化成总体带宽利用率的提升。

## 6. Dynamic Multimem Addressing：让交换机认识"动态目的地"

核心挑战：一个 multimem packet 里只携带一个地址，怎么表示"多个目标 GPU 上各自不同的内存位置"？

作者的关键洞察：Dispatch 是 AllGather 的动态变体。两者都有一个跨 GPU 统一的逻辑索引（algebraic index），区别只是 Dispatch 只发给部分 GPU。因此 packet 里可以继续带 algebraic index（逻辑位置），再加一个轻量的 target expert list（目标列表），让每个目标 GPU 在本地做 algebraic→layout 的地址映射。

如下图，两种寻址方案的对比——explicit addressing（把所有目标地址塞进 packet）vs dynamic multimem addressing（只带一个 algebraic index + target list）：

![两种寻址方案对比](assets/fig5_consideration.png)

explicit addressing 的问题：每个目标地址都要占 header flit，8 个目标时 payload efficiency 从 80% 降到 69%；而且发送方还得预先计算每个目标 GPU 的 remote 内存地址，额外同步开销 5%+。dynamic multimem addressing 更优雅：只带一个 algebraic index，target GPU 自己管自己的内存映射。

整套系统的硬件架构如下图：

![DySHARP 系统架构设计](assets/fig9_architecture.png)

三个组件各司其职：

**源 GPU（Source GPU）**：SM 的 LSU 扩展了一个 MultimemQ，专门处理 `dymultimem.*` 指令，先从 shared memory 取出 target list，再把完整的 request packet 发出去。

**交换机（Switch）**：Route 模块按 target list 做转发和复制；对于 `dymultimem.ld_reduce`（Combine），内置 Reduction Logic 负责等所有 response 到齐后聚合，只把最终结果发回源 GPU。

**目标 GPU（Destination GPU）**：在 Hub 里引入硬件 Memory Manager，维护一个 AL Table（Algebraic-Layout 映射表）和 AL TLB（缓存），把 algebraic index 翻译成本地 virtual address。

如下图，Memory Manager 在 Dispatch 和 Combine 阶段的工作流：

![硬件 Memory Manager 工作流](assets/fig10_manager.png)

- **Dispatch 阶段**：token 第一次到达时，Manager 给它分配一个 layout block，在 AL Table 里写入映射
- **Combine 阶段**：直接复用 Dispatch 时建好的 AL Table，把 `dymultimem.ld_reduce` 的 algebraic index 翻译成 layout index 读取数据

硬件开销：Switch 侧额外逻辑 < 0.01mm²（NVSwitch die 面积的 0.1%），GPU 侧总计 0.198mm²（H100 die 面积的 **0.024%**），基本无感。

## 7. Token-Centric Kernel Fusion：让依赖链细到 token 粒度

即使 dynamic multimem addressing 把流量砍掉近 50%，如果 Dispatch 和 Combine 依然串行，性能提升依然有限——in-switch multicast 减的是 GPU→Switch 流量，in-switch reduction 减的是 Switch→GPU 流量，两个方向各减一半，各自的"未减那侧"仍然是瓶颈。

解法：**让 Dispatch 和 Combine 并发执行**。但这里有数据依赖：

```
Dispatch → GEMM-1 → GEMM-2 → Combine
```

token-centric 的做法：**不等整个 operator 结束**，只要某个 token（或 tile）的 Dispatch 完成了，GEMM-1 就立刻处理这个 tile；GEMM-2 结果出来后，立刻发起 Combine。这样 Dispatch 和 Combine 就能重叠起来。

如下图，token 级别的依赖链和 SM 分区方案：

![Token 级数据依赖链与 SM 分区](assets/fig12_dependency.png)

这套细粒度 readiness 追踪靠的是 Token Tracker，包含三张表：

- **Tile Status (TS) Table**：追踪每个 tile 的 Dispatch→GEMM-1 和 GEMM-1→GEMM-2 的 readiness
- **Token ID (TID) Table**：记录每个 tile 里的 token ID，GEMM-2 完成后通知源 GPU
- **Output Readiness (OR) Table**：源 GPU 侧追踪每个 token 的 top-k expert 输出是否都到齐

如下图，Token Tracker 的完整架构：

![Token Tracker 架构](assets/fig13_tracker.png)

调度器侧：SM 被分成 4 组分别跑 Dispatch、GEMM-1、GEMM-2、Combine；用 megakernel + persistent TB 绕过硬件 TB scheduler，实现 readiness-gated 的细粒度调度。

最终效果：Dispatch（GPU→Switch 主导）和 Combine（Switch→GPU 主导）真正并发，两个方向互补的非对称流量叠加，把理论流量削减变成实际带宽利用率提升。

## 9. 我的看法

这篇工作有几个点值得细品：

**问题切入点很准**。MoE 通信优化已经有一堆工作了，大家都在 overlap 上卷。这篇从"根本上流量就多了"入手，这个视角其实一直都在，只是大家不去碰，因为交换机侧的改造太重了。

**两技术的"组合必要性"论证是亮点**。不是堆 feature，而是把为什么 A 不够、为什么 B 不够、为什么 A+B 才行讲得很清楚。这种系统性的设计逻辑比单点优化有意思得多。

**硬件开销控制得很好**。GPU 侧 0.024%、Switch 侧 0.1% 的面积开销，是在现有 multimem 框架上做最小侵入式扩展，没有推倒重来。

当然也有值得关注的限制：

- 实验全在模拟器上（BookSim2 + Accel-Sim），不是真机，虽然 calibration 误差 <6%
- 落地需要 NVIDIA 在 ISA + 微架构 + Switch 多层同时改动，推进不是小事
- topk 越大收益越大，topk=1 这类极端稀疏场景效果有限

不管怎么说，**把 in-switch computing 从静态 collective 扩展到动态 MoE 通信**，这是整个行业面临的核心矛盾之一，这篇工作是个很有价值的 first step。
