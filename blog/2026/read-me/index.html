把 Dense LLM 变成 MoE 还能推理提速？NeurIPS 2024 Read-ME 做到了

把 Dense LLM 变成 MoE 还能推理提速？NeurIPS 2024 Read-ME 做到了

原文：Read-ME: Refactorizing LLMs as Router-Decoupled Mixture of Experts with System Co-Design

1. 前言

做 MoE 推理优化的牛马应该对这个场景不陌生：

你好不容易把 Mixtral-8x7B 或者 DeepSeek-MoE 部署起来，满心期待地等着”稀疏激活带来的效率红利”落地，结果一跑推理，GPU 显存占用跟 dense 模型差不了多少，延迟也没降多少。

为什么？

因为MoE 的优势是推理时只激活少数 expert，但落地被”全量加载 + 低效调度”的系统设计给卡死了。

这也是整个 MoE 推理领域的核心矛盾之一：模型架构设计和系统推理策略之间的脱节。

今天想聊的这篇 Read-ME（NeurIPS 2024）正面刚了这个问题，而且是从算法和系统两侧同时动刀——它不只是在模型结构上做文章，而是提出了一套 algorithm-system co-design 的框架，实现了在更小模型规模下更好的精度，同时端到端延迟也降下来了。

2. 问题出在哪

先交代下背景，MoE 落地有两个经典难题：

第一个：从头训练 MoE 太贵了。

Mixtral、DeepSeek 这些模型，要么是专门设计的 MoE 架构从头预训练，要么是对已有 dense 模型做 “upcycling”（把 dense FFN 直接复制成多个 expert，然后再训练）。无论哪条路，都要消耗海量 token。对大多数研究者来说，这条路根本走不通。

第二个：传统 layer-wise router 设计在系统层面效率很差。

这是 Read-ME 真正重点解决的问题，也是我觉得最有意思的观察。

现在主流的 MoE（Mixtral、OpenMoE 等）的 router 是 layer-wise 的：每一层都有独立的 router，每层的 expert 选择要等上一层的 hidden state 出来之后才能决定。也就是说，直到 token 真正跑到第 l 层，系统才知道这个 token 要用哪个 expert。

这带来两个系统上的直接问题：

  1. 没法提前 prefetch expert：expert 权重可能分散在不同 GPU/机器上，等到运行时才决定要哪个 expert，来不及预加载，要么 OOM 全加载，要么边算边取等 I/O。

  2. batching 效率极差：一个 batch 里不同 token 需要不同 expert，得等每个 token 都完成当前层计算才能到下一层，GPU 利用率直接拉稀。这里有个很离谱的数据：Mixtral-8x7B 在 batch size 为 56.8 的情况下，平均每层激活的 unique expert 数量是 7.63 个（总共才 8 个），等于说稀疏激活的优势几乎全丢了。

如下图，可以看到 unique expert 数量和平均延迟几乎线性相关——expert 越分散，latency 越高：

3. Read-ME 的核心 idea

Read-ME 的解法分两个层面：算法层 负责把 dense 模型变成高质量的 MoE，系统层 利用算法侧的新特性做推理加速。

3.1 算法侧：Pre-gating Router 解耦

Read-ME 首先在 Llama-2-7B-chat 这类预训练 dense 模型上做 MoE 化，不从头训练，只用 1B tokens 做 continual tuning，代价极小。

具体是怎么提取 expert 的？

基于激活稀疏性（activation sparsity）。Dense LLM 里的 FFN，不同数据域对应的激活 channel 是不同的（科学文章、小说、代码、QA 各自激活不同的神经元），把这些”domain-specific”的 channel 组合成独立的 expert，就得到了领域对齐的 MoE 结构。

这个过程用论文里的话说叫 “Domain-Aware Expert Construction”，本质上是在 activation magnitude 上做 top-k channel 选择，再用一个 mask 矩阵把 FFN 参数切分成 N 个 expert，每个 expert 大约是原始 FFN 参数量的一半（$d \approx D/2$）。

关键创新在 router 设计。

作者发现了一个很有意思的现象：相邻层之间的 expert 选择高度相关。用 Mixtral-8x7B 的 30、31 层实验，转移矩阵几乎是行稀疏的——也就是说，只要知道第 30 层选了哪个 expert，第 31 层几乎是确定性地也会选同样的 expert（超过 70% 的概率）。

如下图可以看到：层间 expert 选择的转移矩阵呈现明显的行稀疏模式（每行只有少数几个非零项），以及相邻层之间的 mutual information 也非常高：

这意味着什么？Layer-wise router 大量在做重复工作，我完全可以用一个统一的 router 在推理最开始就预测所有层的 expert 选择。

Read-ME 把这个 router 叫做 pre-gating router，它和 MoE backbone 完全解耦。结构很简单：一个 1-layer transformer block，只有 18M 参数（可以忽略不计的 overhead）。它在 token 进入 MoE 之前就跑完，一次性输出所有层的 expert 路由决策。

训练的时候用一个 routing distillation loss（KL 散度），让 pre-gating router 的选择和原始 dense 模型里的激活稀疏模式保持对齐，监督信号很自然。

3.2 系统侧：Expert-aware 推理优化

Pre-gating router 的解耦带来了推理系统的三个优化机会：

Fine-grained Expert Prefetching

知道了接下来要用哪些 expert，就能把加载 expert 权重和计算当前层的 forward pass 完全 overlap 起来——算第 i 层的时候，后台同步加载第 i+1 层需要的 expert，完全 hide I/O latency。

Belady-inspired Caching

Expert cache 是个经典问题，以前的策略是 LRU 或者基于激活稀疏性的启发式。

Read-ME 这里做了个很漂亮的 insight：Belady 算法（替换”未来最晚被访问”的那个 cache item）是理论上最优的离线 cache 替换算法，但在普通场景里你不知道”未来”。而有了 pre-gating，你恰好知道未来每个请求会用哪个 expert！

所以 Read-ME 直接实现了近似 Belady 策略，论文也证明了在 pre-gating 下这个 caching 策略是 provably optimal 的。

Expert-aware Batching

Pre-gating 让系统在 batch 组建时就知道每个 token 要用哪些 expert，可以把激活相同 expert 的 token 分到同一个 batch，最大化单个 expert 服务多个 token 的机会，从根本上减少每个 batch 里 unique expert 的数量。

如下图展示了传统 layer-wise MoE 和 Read-ME 的推理流程对比——传统方案 batching 要等到所有层都计算完才能做，而 Read-ME 在请求到达时就能基于 pre-gating 的预测立刻分批：

4. 效果怎么样

4.1 模型质量

用 Llama-2-7B-chat 做 base，提取出来的 Read-ME 模型参数量是 4.7B-17B（4.7B activated parameters），训练代价 只用了 1B tokens，而 Pythia 用了 300B、Llama-2 用了 2T。

在 MMLU、Hellaswag、WinoGrande、ARC-E、ARC-C、LogiQA、CoQA 等 benchmark 上的综合表现：

方法 参数量 训练成本 MMLU avg
Sheared-Llama 2.7B 50B 26.4% 53.2%
Pythia 2.8B 300B 26.9% 48.3%
Open-Llama-v2 3.4B 1T 25.7% 49.3%
OpenMoE 2.1B-8B 1.1T 26.2% —
Read-ME 4.7B-17B 1B 38.9% 55.5%

用 1B tokens 的训练代价，MMLU 比所有 baseline 高出一截，平均综合性能也是最强的。本质上还是 Llama-2-7B-chat 预训练带来的底子，continual tuning 只是把 dense 知识迁移到 MoE 结构里。

（细思极恐的点：Open-Llama-v2 用了 1T tokens 才训到 25.7% MMLU，而 Read-ME 1B tokens 就做到 38.9%……这就是 pretrain-then-convert 策略的威力。）

4.2 推理速度

和 OpenMoE + Llama-2-7b 对比，系统优化的效果：

  • 平均 end-to-end latency 降低 6.1%
  • tail latency（p95）改善 10%

这些数字在 batched inference 场景下是实实在在的提升，而且算法和系统 co-design 的好处是两侧的收益可以叠加。

5. 我的一些看法

这篇工作让我觉得最有意思的地方不是最终的数字，而是那个关于层间 router 冗余性的观察。

大家做 MoE 时候默认每层都需要独立的 router，没人细想过这个设计是不是多余的。Read-ME 用数据说话，发现相邻层 router 的 mutual information 其实很高——换句话说，router 学到的知识在跨层之间是共享的，layer-wise 设计是过设计（overdesign）。

这个 insight 很有迁移价值：如果我们在一开始就设计 MoE 架构（而不是从 dense 转换），是不是也可以直接用共享 router？参数量少了，推理也更友好。

另外值得注意的是，这套框架的适用条件是”从预训练 dense 模型转换 MoE”，需要原始模型本身有足够强的激活稀疏性，才能做出质量够好的 expert 划分。如果原始模型的 FFN 激活很稠密，这条路可能就走不通。

不管怎样，algorithm-system co-design 这个思路是对的，光在模型架构上优化，忽视系统层面的适配，落地效果必然打折。Read-ME 这篇给出了一个完整的闭环方案，也顺手证明了 pre-gating 的 caching 策略是最优的，这个理论保证在工程实践里是很加分的。
