LLM 推理为什么要用 CUDA Graph？从原理到 vLLM 的实际实现

July 08, 2026

2026   ·   LLM   推理加速   CUDA   vLLM   工程优化  

LLM 推理为什么要用 CUDA Graph？从原理到 vLLM 的实际实现

1. 一个让人意想不到的性能杀手

假设你手里有一张 H100，跑 LLM 的 decode 阶段。GPU 很贵，期待它大部分时间在满负荷运算。

结果一 profile，GPU 计算利用率只有 30-40%，剩下的时间在空转。

钱没花在刀刃上。原因是什么？

大模型的一次前向传播，不是一个大 kernel，而是几十上百个小 kernel 的串行组合：RMSNorm、RoPE embedding、attention QKV projection、softmax、FFN gate/up/down projection、residual add……每一个都是独立的 CUDA kernel。

每次调用一个 CUDA kernel，CPU 都要做一次 kernel launch：构造参数、通知 driver、把任务推进 GPU 执行队列。这个操作本身要花 5-20 µs。

单个 kernel 的 launch overhead 看起来不多。但 32 层的模型，每层大概十几个 kernel，一次前向就是几百个 kernel，launch overhead 累计下来就是几毫秒。

Decode 阶段的 memory-bandwidth bound 场景下，总时间可能就 10-20ms，几毫秒的 launch overhead 占比 20-30%。GPU 有将近三分之一的时间在等 CPU 一遍遍地发任务。

CUDA Graph（CG）就是用来解决这件事的。

2. CUDA Graph 是什么：录制一次，replay 无数次

CG 的思路出奇地简单：

第一次推理时，把这几百个 kernel 的执行序列录制下来，生成一张 DAG（有向无环图）：节点是 kernel，边是依赖关系（谁要等谁完成才能开始）。

之后每次推理，直接 replay 这张图——一次 CPU 调用，整张图在 GPU 上执行，不再需要逐个 launch kernel。

几百个 kernel 的 launch overhead，压缩成了一次图 replay 的调用开销。

3. 最小可运行 demo

先看一个最简单的例子，感受一下 capture 和 replay 怎么写：

import torch
import time

device = "cuda"

# 随便搭一个小 MLP 代替 LLM 的一层
model = torch.nn.Sequential(
    torch.nn.Linear(2048, 2048),
    torch.nn.ReLU(),
    torch.nn.Linear(2048, 2048),
).to(device).half()

# ⚠️ 关键：输入 tensor 的地址在整个生命周期里不能变
# 提前分配好，后续就地更新值
x = torch.randn(8, 2048, device=device, dtype=torch.half)

# Step 1：warmup（确保 kernel 已编译、显存已分配）
with torch.no_grad():
    for _ in range(3):
        _ = model(x)

# Step 2：录制
g = torch.cuda.CUDAGraph()
with torch.no_grad():
    with torch.cuda.graph(g):
        y = model(x)   # 这次不真正执行，捕获进 DAG

# Step 3：推理时，就地更新输入，然后 replay
new_input = torch.randn(8, 2048, device=device, dtype=torch.half)
x.copy_(new_input)    # 地址不变，值变了
g.replay()            # 整图在 GPU 上跑，无 kernel launch overhead
print(y.shape)        # 输出地址也是录制时固定的那个

运行对比：

# 对比 eager 和 CG 的推理延迟
N = 1000

# eager 模式
torch.cuda.synchronize()
t0 = time.time()
with torch.no_grad():
    for _ in range(N):
        x.copy_(torch.randn_like(x))
        _ = model(x)
torch.cuda.synchronize()
print(f"Eager: {(time.time()-t0)*1000/N:.2f} ms/iter")

# CG 模式
torch.cuda.synchronize()
t0 = time.time()
for _ in range(N):
    x.copy_(torch.randn_like(x))
    g.replay()
torch.cuda.synchronize()
print(f"CG:    {(time.time()-t0)*1000/N:.2f} ms/iter")

典型结果：在几百个小 kernel 的场景下，CG 比 eager 快 20-40%。

4. CG 最关键的约束：地址固定，拓扑静态

约束 1：tensor 地址不能变

CG 录制的不是 tensor 的”值”，而是每个 kernel 操作的”内存地址”。replay 时 GPU kernel 直接读写这些固定地址。

这意味着：

  • 不能在 replay 时动态 torch.zeros(...) 或 torch.randn(...) 创建新 tensor
  • 输入必须提前分配好，每次推理只能就地 copy_() 更新值
  • 输出 tensor 的地址也是固定的，读结果直接读那个 tensor

约束 2：图的拓扑不能变

录制的 DAG 节点集合和依赖关系在 replay 时完全不能变——不能多跑一个 kernel，不能少跑一个。

这条约束直接排除了一些操作：

  • Python if 分支（会让 capture 只走一条路，replay 时永远走那条）
  • 动态形状的 kernel（batch size 变了，kernel 的 grid 参数就变了）
  • 早退（early exit，运行时决定在哪层停）

5. vLLM 怎么用 CUDA Graph

理解了原理，来看真实系统怎么解决这些约束。vLLM 的 CG 实现是整个推理框架的核心优化之一，代码集中在：

  • vllm/compilation/cuda_graph.py — 图的录制/replay 逻辑
  • vllm/v1/cudagraph_dispatcher.py — 运行时选哪张图
  • vllm/v1/worker/gpu_model_runner.py — 输入 buffer 管理和 capture 流程

5.1 固定地址的输入 buffer

对于”tensor 地址不能变”这个约束，vLLM 的解法是：在 runner 初始化时一次性预分配所有输入 buffer，整个服务生命周期里这些 buffer 的地址都不变。

# vllm/v1/worker/gpu_model_runner.py（简化版）
class GPUModelRunner:
    def _init_buffers(self):
        # 预分配固定地址的输入 buffer
        self.input_ids = self._make_buffer(self.max_num_tokens, dtype=torch.int32)
        self.positions  = torch.zeros(self.max_num_tokens, dtype=torch.int64, device=self.device)
        self.query_start_loc = self._make_buffer(self.max_num_reqs + 1, dtype=torch.int32)
        self.seq_lens   = torch.zeros(self.max_num_reqs, dtype=torch.int32, device=self.device)
        # ... 其余 buffer 同理

每次推理前，往这些固定地址的 buffer 里 copy 新数据，然后 replay 图。图读到的始终是这些固定地址，约束满足。

5.2 批大小分桶（Batch Size Bucketing）

这是解决”图的拓扑不能变”约束里最有意思的工程设计。

问题：CG 录制时 batch size 是固定的，但实际推理的 batch size 一直在变（每轮 decode，可能有请求完成、有新请求进来）。如果每个 batch size 都录制一张图，理论上要录制 1 到 max_batch_size 这么多张，消耗大量显存和录制时间。

vLLM 的解法：只录制几个”档位” batch size 的图，中间值 padding 到上一档。

默认录制的档位（cudagraph_capture_sizes）：

[1, 2, 4, 8, 16, 24, 32, 48, 64, 80, 96, 112, 128, 160, 192, 224, 256]

运行时的映射逻辑（cudagraph_dispatcher.py）：

def _compute_bs_to_padded_graph_size(self):
    capture_sizes = [1, 4, 8, 16, 32, 64]   # 简化示例
    max_size = 128

    self._bs_to_padded: list[int] = [0] * (max_size + 1)

    for end, start in zip(
        capture_sizes + [max_size + 1],   # [4, 8, 16, 32, 64, 129]
        [0] + capture_sizes,              # [0, 1, 4,  8, 16,  32]
    ):
        for bs in range(start, end):
            # bs == start 时映射到自己，否则映射到下一个档位上界
            self._bs_to_padded[bs] = start if bs == start else end

映射结果（举例）：

实际 batch size padding 到 使用的图
1 1 bs=1 的图
2, 3 4 bs=4 的图
4 4 bs=4 的图
5-7 8 bs=8 的图
8 8 bs=8 的图

padding 的 slot 里填 padding token，对应位置的输出被丢弃。显存里预分配的是 max_batch_size 大小，padding 不需要额外开销，只是”有效 token 数”变了。

5.3 CUDAGraphWrapper：录制和 replay 的核心逻辑

# vllm/compilation/cuda_graph.py（关键路径简化）
class CUDAGraphWrapper:
    def __call__(self, *args, **kwargs):
        key = self._make_key(args)   # 根据 batch size 等信息生成 key
        entry = self.entries.get(key)

        if entry is None or entry.cudagraph is None:
            # 第一次见到这个 key，录制
            cudagraph = torch.cuda.CUDAGraph()
            with torch.cuda.graph(cudagraph, pool=self.graph_pool):
                output = self.runnable(*args, **kwargs)
            entry = CUDAGraphEntry(cudagraph=cudagraph, output=output)
            self.entries[key] = entry
        else:
            # 已录制过，直接 replay
            entry.cudagraph.replay()

        return entry.output

注意 graph_pool 参数——vLLM 让所有图共用同一个显存池（get_global_graph_pool()），大 batch 图先录制，小 batch 图录制时可以复用大图分配过的显存块，减少总显存占用。

5.4 KV cache 怎么处理

KV cache 是推理里最棘手的部分——它随着序列生成不断增长，物理内存块的位置会变，和 CG”地址不变”的约束直接冲突。

vLLM 的解法是 slot mapping 间接层：

逻辑位置（token 在序列里的位置）
      ↓  slot_mapping（一个可就地更新的 tensor）
物理 KV block 地址

attention kernel 里，读 KV 时不直接用序列位置作为 cache 地址，而是先查 slot_mapping 这个 tensor 找到实际的物理 block，再去读。

slot_mapping 是预分配的固定地址 tensor，每次推理前就地更新它的内容（哪些逻辑位置映射到哪些物理 block）。CG 捕获的是 kernel 读 slot_mapping 的行为，不是 KV cache 的实际地址。这样 KV cache 的物理位置随时可以变，只要更新 slot_mapping 就行，图本身不需要重新录制。

6. 录制成本：为什么要先 warmup，什么时候重新 capture

录制本身是有成本的，这也是使用 CG 时常见的踩坑点。

为什么要 warmup

CUDA kernel 第一次运行时有编译开销（JIT 编译、PTX 转 SASS）。如果在这个阶段录制，录进去的 kernel 和后续 replay 的 kernel 可能版本不一致，甚至崩溃。vLLM 在 capture 前都会跑 3-5 次 warmup，确保所有 kernel 都编译完成、显存分配稳定。

capture 发生在什么时候

vLLM 启动时，按从大到小的顺序把所有档位的图都 capture 好：

# gpu_model_runner.py 的启动流程（简化）
def _capture_all_cudagraphs(self):
    # 大 batch 先 capture，显存池建好了，小 batch capture 时可以复用
    for batch_size in sorted(capture_sizes, reverse=True):
        self._capture_single_cudagraph(batch_size)

大 batch 优先是为了显存复用：CUDA graph pool 机制允许后来录制的图”蹭”已有图用过的显存区域，减少总占用。

什么时候不能用 CG

prefill 阶段（处理 prompt）的序列长度每次都不一样，且往往很长——不适合按 batch size 分桶，vLLM 的 prefill 默认走 eager mode，只有 decode 阶段用 CG。

7. 实际收益有多大

几个有代表性的数据点（来自 vLLM 的 profiling 和相关论文）：

  • Llama-3-8B，decode，batch=1：CG 相比 eager 快约 20-30%，绝对值从 ~15ms 降到 ~11ms
  • Llama-3-70B，decode，batch=8：CG 加速约 15-20%（模型越大，kernel 越耗时，launch overhead 占比越小，收益相对减少）
  • 小模型（1-3B），decode，batch=1：CG 收益最显著，launch overhead 占比最高，加速可达 40-50%

8. 踩坑备忘

坑 1：capture 时不能有 Python 控制流依赖 GPU 结果

# ❌ 这样会出问题：capture 时走了 True 分支，replay 时永远走 True
with torch.cuda.graph(g):
    x = model(input)
    if x.max() > threshold:   # x.max() 是 GPU 结果，capture 时已经固定
        x = postprocess(x)

解法：把条件判断移到 graph 外面，用两张不同的图，或者把条件改成 GPU tensor 操作（torch.where）。

坑 2：capture 时动态分配的 tensor 在 replay 时地址会变

# ❌ 每次 replay，torch.zeros 会分配新地址
with torch.cuda.graph(g):
    mask = torch.zeros(batch_size, device="cuda")   # 地址在 capture 时固定
    # 但 replay 时这行代码不会重新执行，mask 还是 capture 时那块内存

解法：把所有需要动态值的 tensor 提前分配好，replay 前就地 fill_() 或 copy_()。

坑 3：显存不够

每张图都要在 GPU 显存里占一块 activation buffer（录制时分配的临时 tensor）。多档位 × 2（warmup + capture）会多占几 GB。vLLM 的 graph_pool 机制缓解了这个问题，但 OOM 还是可能发生——遇到时调小 max_cudagraph_capture_size 或减少录制档位数。

9. 一句话总结

CUDA Graph 做的事情就一件：把”反复 launch 同样一批 kernel”变成”录一次图，replay N次”，把 N 次 CPU kernel launch overhead 压缩成一次 replay 开销。代价是 tensor 地址必须固定、图的拓扑必须静态。

vLLM 用三个工程设计把这两个约束处理掉：

  1. 预分配固定地址的 buffer，推理时就地更新内容
  2. batch size 分桶 + padding，有限几张图覆盖所有实际 batch size
  3. slot mapping 间接层，让 KV cache 的物理地址可以随时变，图不需要重录

理解了这三个设计，vLLM 里 CG 相关代码的逻辑就基本清楚了。欢迎评论区交流。
