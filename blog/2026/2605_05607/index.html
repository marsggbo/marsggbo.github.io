MoE 推理训练又双叒叕被卡通信了？DySHARP 用交换机内计算直接干掉 50% 冗余流量

原文：Accelerating MoE with Dynamic In-Switch Computing on Multi-GPUs

1. 前言吐槽

你有没有想过，训练 DeepSeek-V3 这种 MoE 模型时，GPU 的通信线路上到底跑了多少无用数据？

我去仔细看了一下，结果有点细思极恐——近 50% 的通信流量是纯冗余的，白白占着带宽，什么有价值的事也没干。

更离谱的是，NVIDIA 其实在 Hopper 时代就搞了一个叫 NVLink SHARP（NVLS）的东西，本来可以在交换机里做计算来消掉这些冗余，但结果因为 MoE 的通信模式是”动态的”，而 NVLS 只支持”静态的”，所以直接 GG。

这篇 ISCA 2026 的工作（DySHARP）就是来填这个坑的。今天想和大家聊聊这篇工作到底解决了什么真问题，以及它的两个核心设计有多精妙。

2. 核心问题：MoE 的通信瓶颈到底有多严重

先交代下背景。MoE（Mixture-of-Experts）结构的精髓就是稀疏激活：每个 token 只走 top-k 个 expert，计算量大幅减少。但麻烦来了——在多 GPU 训练时，expert 分散在不同 GPU 上（expert parallelism，EP），每个 token 就要通过 inter-GPU 通信找到自己对应的 expert，跑完计算再把结果传回来。

这两个操作有名字：Dispatch（把 token 发出去）和 Combine（把 expert 输出汇总回来）。

问题有多严重？在 DeepSeek-V3 的实验里，通信耗时占 MoE layer 执行时间的 70.4%。随着 Blackwell、Rubin 一代比一代算力猛，GPU 算得越来越快，这个比例只会更高——算力把通信彻底甩开了。

如下图，整体 MoE 执行流程（Dispatch → GEMM-1 → GEMM-2 → Combine），通信把计算夹在中间，活生生被卡住。图中 (a) 是 Expert Parallelism 的基本架构，(b) 揭示了 Dispatch 和 Combine 中各自的通信冗余，(c) 是 in-switch computing 理论上可以解决这些冗余的方式：

3. 冗余在哪？一个 token 被传了好几次

这里有个细节很关键，也是整篇工作的出发点。

Dispatch 阶段的冗余：一个 token 可能需要发送到多个 GPU（top-k 的 k 不是 1），那么源 GPU 要把同样的数据发 k 次，每次走一条链路。比如 Token B 需要发给 GPU2 和 GPU3，GPU0 就得传两遍完全一样的数据。

Combine 阶段的冗余：各 GPU 上算完的 expert 输出，要分别发回给源 GPU，这些数据是”可聚合”的，但现在是分多条链路分别发。

在 DeepSeek-V3 + 32-GPU GH200 NVL32 的模拟环境里，当激活专家数 ≥ 8 时，冗余流量占总通信流量近 50%。理论上，如果能把这些冗余干掉，通信速度可以接近 2× 提升。

4. NVLS 为什么搞不定？

NVIDIA 早就想到了交换机内计算（in-switch computing）这个方向，NVLS 就是为此设计的。原理很直接：

  • in-switch multicast：GPU0 只往交换机传一次数据，交换机自己负责分发给 GPU2 和 GPU3，消除 Dispatch 冗余
  • in-switch reduction：GPU2 和 GPU3 的结果在交换机内部直接聚合，再发回 GPU0，消除 Combine 冗余

听起来完美对吧？但 NVLS 的设计是为静态 collective 操作（AllGather、Reduce-Scatter）量身定做的，有两个硬性约束：

  • Fixed target sets：所有 token 都和同一组 GPU 通信（所有 GPU）
  • Symmetric addressing：token 在每个 GPU 上的内存偏移量一致

MoE 的 Dispatch/Combine 偏偏不满足这两条：

  • Varying targets：Token A 可能发往 GPU {0, 1, 2}，Token B 发往 {2, 3}，每个 token 的目的地不同
  • Asymmetric addressing：每个 GPU 独立管理本地内存，同一个 token 在不同 GPU 上的 layout offset 不一样

如下图，静态通信 vs 动态通信的模式对比：

有人可能想到一个 workaround：把 MoE 的 Dispatch 转换成 AllGather，Combine 转换成 Reduce-Scatter，强行套用 NVLS。问题是这会产生大量无用流量——对不需要某个 token 的 GPU 也发过去，白费带宽。实测这个方案引入了 340% 的无用流量，彻底把 in-switch 省下来的那点好处给冲掉了。

5. DySHARP：两个技术，缺一不可

这篇工作提出了 DySHARP，核心是两个技术的组合拳：

① Dynamic Multimem Addressing（动态多目标寻址）：让交换机支持动态通信 ② Token-Centric Kernel Fusion（以 token 为中心的算子融合）：把流量削减真正转化成速度提升

为什么两个缺一不可？下图完整呈现了 6 种配置的时间轴对比，从 (a) 到 (f) 逐步演进：

  • (a) 无 overlap 的 baseline（DeepEP）：Dispatch → GEMM-1 → GEMM-2 → Combine 串行执行，通信全部暴露；
  • (b) 基础 overlap baseline（COMET）：Dispatch-GEMM-1 和 GEMM-2-Combine 可以 overlap，但通信量本身没变；
  • (c) DySHARP-Basic（dynamic multimem addressing，无 overlap）：GPU→Switch 和 Switch→GPU 各减了一半流量，但方向不对称——单独执行 Dispatch 或 Combine 时，被未削减那侧拖住，不能直接转化成速度；
  • (d) DySHARP-COMET（DySHARP-Basic + 基础 overlap）：流量少了但仍不对称，Dispatch 和 Combine 各跑各的，瓶颈没消除；
  • (e) 仅 Kernel Fusion（无 in-switch computing）：Dispatch 和 Combine 并发了，但流量没少，相比 COMET 无改善；
  • (f) 完整 DySHARP：两者结合——Dispatch 主导 GPU→Switch、Combine 主导 Switch→GPU，并发执行使互补流量叠加，双向带宽都被充分利用，才把 50% 流量削减真正变成了速度提升。

6. Dynamic Multimem Addressing 怎么设计的

核心挑战：一个 multimem packet 里只带一个地址，怎么表示”多个不同 GPU 上的不同 offset”？

作者的洞察是：Dispatch 其实是 AllGather 的动态版本——algebraic index（逻辑张量里的 token 位置）跨 GPU 是一致的，但 layout index（实际内存里的位置）因为只有部分 GPU 接收而各不相同。

所以设计就变成：

  • Packet 携带：一个 multimem address（algebraic index）+ 一个 target expert list（目的地列表）
  • Switch 负责：根据 target list 做转发和 replication
  • 目标 GPU 负责：用一个硬件 memory manager，把 algebraic index 映射到本地的 layout index（即实际物理地址）

如下图，详细的架构设计和工作流程：

为了支持这套寻址机制，作者对 NVLink packet format、ISA、GPU SM 的 LSU，以及 NVSwitch 都做了扩展：

Packet Format：在原来的 flit0 里，把 64-bit 地址替换成 48-bit multimem address + 1-bit stage（Dispatch/Combine）+ 15-bit target count，后续 flit 编码 target expert ID 列表。

ISA 扩展：引入 dymultimem.st（Dispatch 用，multicast）和 dymultimem.ld_reduce（Combine 用，reduction）两条指令，比原来的 multimem 指令多两个寄存器：一个存 target count，一个存 target list 的基地址。

目标 GPU 的 Memory Manager：在 GPU Hub 里引入 AL Table（Algebraic-Layout 映射表）和 AL TLB（缓存），把到来的 multimem address 翻译成本地 virtual address 再访问内存。

硬件开销评估结果很轻量：Switch 侧额外逻辑不到 0.01mm²，GPU 侧总共 0.198mm²，约为 H100 die 面积的 0.024%，基本无感。

7. Token-Centric Kernel Fusion 怎么做的

即使有了 dynamic multimem addressing 把流量砍掉 50%，如果 Dispatch 和 Combine 还是串行执行，性能提升依然有限。

本质上还是那个不对称问题：

  • Dispatch 削减了 GPU→Switch 方向的流量，但 Switch→GPU 方向（Combine 那侧）没减
  • Combine 削减了 Switch→GPU 方向的流量，但 GPU→Switch 方向（Dispatch 那侧）没减

解法：让 Dispatch 和 Combine 并发执行，两个方向的互补流量叠加，双向带宽都被充分利用。

但并发执行 Dispatch 和 Combine 不是说一句话的事，因为中间还有 GEMM-1 和 GEMM-2（expert 计算），存在数据依赖：

Dispatch → GEMM-1 → GEMM-2 → Combine

一个 token 必须先 Dispatch 到达，才能开始 GEMM-1；GEMM-2 出结果后，Combine 才能发起。

token-centric 的意思是：不等整个 operator 执行完，以 token/tile 为粒度检测 readiness，条件一满足立即执行下一步。

如下图，token 级别的数据依赖链和 SM 分区方案：

为了实现这套细粒度的依赖追踪，作者设计了一个 Token Tracker：

  • Tile Status (TS) Table：追踪每个 tile（一批 token）在 Dispatch→GEMM-1、GEMM-1→GEMM-2 的 readiness
  • Token ID (TID) Table：记录每个 tile 里的 token ID，GEMM-2 完成后用于通知源 GPU
  • Output Readiness (OR) Table：追踪每个 token 的 top-k expert 输出是否都到齐，到齐才触发 Combine

如下图，Token Tracker 的架构设计：

Scheduler 侧：用 SM partition 把 GPU 里的 SM 分成四组分别跑 Dispatch、GEMM-1、GEMM-2、Combine，用 megakernel + persistent thread blocks 绕过硬件 TB scheduler 做细粒度调度，readiness 没达到就阻塞。

最终效果：Dispatch（GPU→Switch 主导）和 Combine（Switch→GPU 主导）真正并发，互补的非对称流量合并，整体带宽利用率大幅提升。

8. 实验效果怎么样

实验在模拟的 GH200 NVL32（32 GPU 系统）上跑，模型参考 DeepSeek-V3 配置（Small/Medium/Large 三档，topk = 8/16/32）。

端到端训练加速（对比各 baseline）：

  • vs DeepEP：最高 2.31×，几何均值 1.93×
  • vs NVLS（静态 workaround）：最高 5.12×，几何均值 3.38×（它产生的无用流量太多了）
  • vs FasterMoE：最高 2.11×
  • vs COMET（SOTA 细粒度 overlap）：最高 1.79×，几何均值 1.59×

  • vs DeepEP：最高 2.77×，几何均值 2.26×

消融实验也很清楚地验证了”两个技术缺一不可”：

  • DySHARP-Basic（只加 dynamic multimem addressing，无 overlap）：流量减了，但由于通信不对称，不能直接转化为速度
  • DySHARP-COMET（加了 basic overlap）：依然受不对称限制
  • Kernel Fusion Only（无 in-switch computing）：相比 COMET 没提升
  • DySHARP（完整方案）：把流量削减和不对称消除都做到位，打通加速的最后一公里

Scalability：随 GPU 数量从 4 到 64 增长，DySHARP 的优势持续扩大，这也说明在更大规模 SuperPOD 上潜力更大。

推理也管用：包括 prefill 和 decode 阶段都有明显提升，不只是训练场景。

其他模型：GPT-OSS-120B 和 Qwen3-235B 上也验证了同样的优势。

9. 我的一些看法

这篇工作我觉得有几个点值得细品：

第一，问题的切入点很准。MoE 通信优化已经有一堆工作了，但大家都在 overlap 上卷，没人从”根本上流量就是多的”这个角度下手。50% 的冗余流量其实一直在那，只是大家不去碰，因为交换机侧的改造太重了。

第二，两个技术的组合是真正的系统性思考。如果只提 dynamic multimem addressing，流量确实少了，但不对称的问题导致加速上不去。单独做 kernel fusion，既然没有流量削减，能做的也就是 overlap，并没有超越 COMET。只有两者结合，互补的非对称流量才能真正被 balance 掉。这个设计逻辑链条是清晰的。

第三，硬件开销控制得很好。GPU 侧 0.024%、Switch 侧 0.1% 的面积开销，说明这套方案在工程上真的是”最小侵入式”扩展，没有推倒重来，而是在现有 multimem 框架上做 dynamic 扩展。

当然也有一些值得关注的 limitation：

  • 实验是在模拟器上跑的（BookSim2 + Accel-Sim），不是真机；虽然用 DGX-H100 做了 6% 误差内的 validation，但现实部署还有未知因素
  • 要真正落地，需要 NVIDIA 在硬件层面支持 dynamic multimem 寻址，这涉及 ISA + 微架构 + Switch 的多层改动，推进起来不是小事
  • topk 越大，收益越大；对于 topk=1 这类极端稀疏场景，可能效果有限

不管怎么说，这个方向——把 in-switch computing 从静态 collective 扩展到动态 MoE 通信——是这个行业整体需要解决的核心矛盾之一，这篇工作是个很有价值的 first step。
