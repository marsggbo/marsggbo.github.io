PiKV：MoE 推理的 KV Cache，也该变稀疏了

原文：PiKV: KV Cache Management System for Mixture of Experts

代码：NoakLiu/PiKV

你有没有想过，MoE 模型明明每个 token 只激活少数几个 expert，为什么真正部署的时候显存还是经常爆？

这个问题比表面看起来更绕一点。MoE 的计算是稀疏的，但 KV Cache 很多时候仍然按 dense 的方式在存、在同步、在跨 GPU 访问。也就是说，forward 的 MLP 部分省下来了，但 attention 这边的历史记忆还在老老实实把账单寄给显存和通信系统。

今天想聊的这篇 PiKV，就是在问一个很系统的问题：

既然 MoE 的计算可以按 expert 稀疏激活，那 KV Cache 为什么不能也按 expert 稀疏管理？

如下图，PiKV 把 KV Cache 不再看成一坨被动存储，而是拆成 routing、compression、scheduling 和 expert-sharded storage 几个模块一起管。

1. 先交代背景：MoE 的稀疏计算，撞上了 dense KV Cache

MoE 的核心卖点很直观：模型参数很多，但每个 token 只走 top-k 个 expert。比如有 16 个 experts，每次只激活 2 个，那 MLP 计算理论上就不用碰剩下 14 个 expert。

这也是为什么 MoE 看起来很适合大模型扩展：参数规模上去了，但单 token 计算量没有线性上去。

但推理系统里还有一个很烦人的东西：KV Cache。

自回归生成时，每生成一个新 token，都要 attend 到前面所有 token 的 Key/Value。context 越长，KV Cache 越大。论文里给了一个很直观的例子：对一个 7B 级别、128K context、16 experts 的 MoE 模型，完整 KV Cache 可以超过 24GB。

这里的关键矛盾是：

MoE 稀疏的是 expert 计算，但很多系统里的 KV Cache 访问仍然是 dense 的。

换句话说，模型说“我这次只想找两个 expert 干活”，但 cache 系统说“不行，历史包袱大家一起背”。这就很离谱。

2. PiKV 的核心思路：让 KV Cache 跟着 expert 一起分片

PiKV 的第一个设计叫 expert-sharded KV storage。

传统做法里，KV Cache 往往按照 token 或 layer 的维度组织，然后在多 GPU / 多节点之间复制、同步或按块搬运。PiKV 换了一个视角：既然 MoE 的路由已经告诉我们某个 token 和哪些 expert 有关系，那 KV Cache 也应该按 expert 维度切开。

论文里的 shard 函数大概长这样：

s(t, e) = (t mod N_tok) xor (e mod N_exp)

这里 t 是 token 位置，e 是 expert id。这个设计不一定要神化，核心不是这个 hash 函数多玄学，而是它想保证两件事：

  1. token 的 KV 能按 expert 分散到不同 shard；
  2. 每个 GPU 不必保存全量 expert 对应的历史 KV。

说人话就是：

不要让每个 expert 都背完整上下文，只让它背自己可能用得上的那部分记忆。

这和 MoE 的精神是一致的。MoE 本来就是“该哪个 expert 干活，就让哪个 expert 干活”；PiKV 进一步说，“该哪个 expert 记，就让哪个 expert 记”。

3. PiKV Routing：不只是选 expert，还要考虑 cache miss

第二个模块是 PiKV Routing。

普通 MoE routing 主要关心 top-k expert 的打分，比如 token 应该发给哪些 expert，负载是否均衡，是否出现 expert collapse 等。PiKV 的 routing 多了一层系统味道：选 expert 时不只看模型分数，还要看 KV Cache 访问代价。

论文列了多种 routing 策略：

比如：

  • RB：hash / round-robin，简单但粗糙；
  • RT：TopK softmax，标准 MoE 风格；
  • RLB：加 load balance；
  • RP：cache-aware 的 PiKVRouter，会惩罚 cache miss；
  • RH：hierarchical routing，先粗选再细选。

最值得注意的是 RP 这类 cache-aware routing。它的味道有点像系统里的 locality-aware scheduling：不是每次都只问“哪个 expert 最懂这个 token”，还要问“找这个 expert 的 cache 代价大不大”。

论文给的复杂度分析也很直接。假设一共有 E 个 experts，每个 query 只路由到 k 个 experts，那么 dense 的 attention / KV fetch 规模类似：

C_dense  = B * L * h * E
C_sparse = B * L * h * k

理论上的计算和 memory traffic relief 都是 E / k。

当然，真实系统里不会这么线性，因为通信、kernel launch、load balance、cache hit rate 都会把理论收益吃掉一部分。但这个公式说明了 PiKV 的设计目标：不是单纯压缩 KV，而是减少“本来就不该访问”的 KV。

4. Compression：KV Cache 不只要少访问，还要少占空间

第三个模块是 PiKV Compression。

KV Cache 压缩这几年已经有很多工作了，比如 PyramidKV、ChunkKV、SVD、量化、FastV 等。PiKV 的定位不是提出一个全新的压缩算法，而是把这些 compressor 放进一个统一的 MoE cache pipeline 里。

论文里把压缩写成：

C(K, V) = (K_hat, V_hat),  d' < d
rho = d / d'

也就是把原来维度为 d 的 K/V 压到 d'，压缩率是 rho。

如果只看一次 step 的读 cache 和 decode 开销，论文给了一个很朴素的式子：

T_read   = 2 * d * k * B / (rho * beta)
T_decode = eta * d * k * B / (rho * gamma)

这里 beta 是 HBM bandwidth，gamma 是 compute throughput，eta 是 decode factor。

这个公式其实很好懂：压缩率 rho 越大，读的数据越少，decode 的数据也越少，所以 latency 下降。但压缩不是白嫖，压太狠之后 reconstruction error 和模型质量会掉，而且 decode 本身也会变成新的瓶颈。

所以我觉得 PiKV 这里更像一个“系统集成框架”：它没有押宝某一种 compressor，而是让 routing、compression、scheduler 可以组合。

这点工程上挺重要。因为不同场景的最优解不一样：

  • 长上下文 summarization 可能更吃 memory；
  • 低延迟 chat serving 更吃 TTFT / TBT；
  • 高吞吐 batch inference 更吃 cache hit 和调度；
  • MoE 模型还要额外考虑 expert load balance。

一个固定策略很难通吃。

5. Scheduling：KV Cache 管理，本质上是“留谁、踢谁”

第四个模块是 PiKV Scheduling。

如果说 routing 决定“这次访问谁”，compression 决定“每份 cache 多大”，那 scheduling 解决的就是最现实的问题：

显存就这么多，到底哪些 KV entry 该留，哪些该踢？

PiKV 把每个 cache page 赋一个 utility score，比如 attention intensity、recency、frequency、age 等，然后按策略做 eviction。它支持 H2O、StreamingLLM、QUEST、LRU、AdaKV、DuoAttention 这类策略。

我比较喜欢论文里这一段 memory 公式，因为它把一个工程问题压成了一个很清晰的设计规则。

设：

  • d：原始 KV hidden size；
  • rho：压缩率；
  • L：每个 expert 的全局 cached tokens；
  • G：GPU 数；
  • S：每个 expert shard 的 circular buffer size；
  • K：scheduler 每个 GPU 保留的 active pages 数。

那么 PiKV 的 per-GPU memory 可以写成：

M_total = 2d / rho * ( L / (G * S) + K * S )

对 S 求导，可以得到最优 buffer size：

S* = sqrt( L / (K * G) )

这个式子的价值不在于它能精确预测所有真实部署场景，而在于它给了一个很好的系统直觉：

buffer 太小，shard 太碎，metadata / 调度 overhead 上来；buffer 太大，保留 page 太多，显存又被吃掉。最优点来自两者的平衡。

这也是系统论文里我比较喜欢的部分：不是只说“我们调了个参数效果好”，而是把参数背后的 trade-off 写出来。

6. 实验结果：收益看起来不错，但要注意来源

这里需要单独说一句：我读的是 arXiv v2。v2 PDF 正文主要给了系统设计、公式和模块说明，完整实验表更多放在 GitHub 的 downstream_tasks/README.md 里。所以这里的实验数字，我会把它当作项目实验结果来看，而不是说“PDF 正文完整展示了这些表”。

根据 repo README，PiKV 在不同模型和组合策略下给了几组结果：

  • repo 首页给出的 summary：PiKV Quantized 相比 Standard MoE，memory usage 从 100% 降到 35%，inference speed 到 2.2x；Enhanced 版本 memory usage 到 30%，speed 到 2.5x；
  • Qwen 14B 实验里，单独使用 PyramidCompressor 可到 2.8x compression、latency 108ms、KV memory 2.1GB；QuantizedCompressor 到 3.4x compression、1.8GB KV memory；
  • LLaMA 2 7B、batch size 512、context 4k 的实验里，baseline KV memory 大约 3.4GB；LoRACompressor / LoRA++ 可以降到 1.1-1.2GB，但 accuracy drop 会到 6-7% 甚至更高；
  • 一些 triple-technique 组合能把 compression 推到 3-4x 以上，但 accuracy drop、KV hit rate、latency 之间会明显 trade-off。

我个人比较关注的不是“最大数字”，而是这个趋势：

PiKV 不是一个单点优化，而是把 routing、compression、scheduler 三个旋钮放到同一个系统里调。

这意味着它能覆盖更多 deployment point，但也意味着评估会变复杂。比如一个配置 memory 最省，但 accuracy drop 可能更大；另一个配置 latency 更好，但 KV hit rate 下降；再一个配置保持质量不错，但压缩收益一般。

这类系统最后真正能不能落地，关键不只是平均指标，而是能不能在真实 serving workload 下找到稳定的 Pareto frontier。

7. 我觉得这篇工作最有意思的点

第一，它抓住了 MoE 推理里一个容易被忽略的不一致性。

大家讨论 MoE 时，经常盯着 expert 权重怎么放、all-to-all 怎么优化、routing 怎么负载均衡。但 KV Cache 这边如果仍然 dense，那 sparse compute 的收益会被 memory 和 communication 吃掉。PiKV 把这个矛盾单独拎出来，是对的。

第二，它把 KV Cache 从“存储问题”改写成“查询驱动的资源调度问题”。

这点和我之前看 Agent memory / KV compression 的感觉一致。KV Cache 不是一块死内存，它更像一个运行时数据库：有查询、有索引、有 eviction、有压缩、有 locality，也有跨节点访问代价。既然是数据库味道的问题，就不能只靠一个简单 LRU 或 fixed window 解决。

第三，它对 MoE 的 sparse routing 和 cache locality 做了耦合。

这点很关键。很多 KV compression 方法默认模型结构是 dense Transformer，只是在 token / layer 维度做筛选。PiKV 直接把 expert 维度也纳入 KV 管理，这会更贴近 MoE serving 的真实瓶颈。

8. 但我也有几个保留意见

首先，论文 v2 目前更像一个系统框架 / living project 的技术报告，而不是一篇实验闭环非常完整的系统论文。很多实验细节、baseline 设置、硬件环境、真实 workload 分布，需要去 repo 里拼起来看。

其次，PiKV 组合了很多模块：routing、compression、scheduler、distributed storage、vLLM integration、DeepSpeed integration。模块多是灵活，但也会带来一个问题：

最后收益到底来自哪个模块？

系统论文里 ablation 很重要。比如 cache-aware routing 单独贡献多少？expert-sharded layout 单独贡献多少？compression 的收益是否可以被现有 KV compression 方法直接拿到？scheduler 是否只是在复用 H2O / QUEST / AdaKV 的收益？这些都需要更细的实验拆解。

第三，MoE serving 真正麻烦的地方不只是 KV Cache，还包括 expert placement、batching、prefill/decode disaggregation、dynamic batching、跨节点 all-to-all、hot expert 长尾等。PiKV 现在的方向和这些问题是相关的，但如果要进生产系统，还需要和 vLLM / SGLang / TensorRT-LLM 这类框架做更深的融合。

论文也承认了一个限制：当前 prototype 评估到 16 GPUs，扩到上百 GPU 的 production cluster 仍然是开放问题。

9. 和之前 KV Cache 工作的关系

如果把最近几类 KV Cache 工作放在一起看，大概可以这么分：

  • vLLM / PagedAttention：解决 KV Cache 内存分页和 serving 管理；
  • H2O / StreamingLLM / QUEST：解决长上下文下哪些 token 值得保留；
  • DroidSpeak / CacheSlide / TokenDance：解决不同模型、Agent、多 prompt 场景下的 KV Cache 复用；
  • PiKV：解决 MoE 架构下 KV Cache 和 expert sparsity 不匹配的问题。

PiKV 的位置挺清楚：它不是“再做一个 KV 压缩算法”，而是把 MoE 结构信息纳入 KV Cache 系统设计。

我觉得这个方向后面会越来越重要。因为 DeepSeek、Qwen、Mixtral 这类 MoE 模型已经证明了 sparse expert 的实用性，但真正大规模部署时，瓶颈一定会从单纯 FLOPs 转向 memory、communication、scheduler。

到那个时候，只让计算稀疏是不够的，memory system 也必须理解稀疏性。

10. 写在最后

PiKV 这篇工作最值得记住的一句话，我觉得是：

MoE 的稀疏性不应该只停留在 expert compute 上，也应该传导到 KV Cache 管理里。

它把 KV Cache 看成一个 query-driven、sparsity-aware、compression-aware 的系统组件，而不是 attention 后面那坨默认存在的显存开销。这种视角是有价值的。

当然，如果按成熟系统论文的标准看，它还需要更强的 end-to-end benchmark、更清晰的 ablation、更真实的 serving workload，以及和主流推理框架更深的实测对比。

但方向本身我挺认可。做 LLM inference 的人都知道，真正卡系统的很多时候不是“模型会不会算”，而是“数据能不能按正确的时间、正确的粒度、正确的路径到达 GPU”。

KV Cache 正是这个问题最集中的地方。

MoE 已经把参数计算变稀疏了，下一步很自然：让记忆也别再傻乎乎地 dense 下去了。
